Portfolio Project 4: Full-Stack Toolkit

Project purpose:
In this project, you'll build a Full-Stack site based on business logic used to control a centrally-owned dataset. You will set up an authentication mechanism and provide role-based access to the site's data or other activities based on the dataset.

Main Technologies:
HTML, CSS, JavaScript, Python + Django
Relational database (recommending MySQL or Postgres)

Project Ideas

You can either choose to build the website based on Project Idea 0 or take inspiration from the other example ideas below.

Project Idea 0
Bring your idea(s) to life, based on providing value to users to address a specific factual or imagined need. Use the relevant project assessment criteria as a guide to the minimum required functionality

Project Example Idea 1
A Restaurant booking System

External user’s goal:
The user would like to book one or more guests for a meal in a restaurant and a particular time and date.

Site owner's goal:
The site owner would like the ability to take online bookings for their eatery.

Potential features to include:

  • Date/Time-based bookings
  • Avoid double bookings
  • Multiple table occupancies
  • Cancellations
  • Menu

Project Example Idea 2
Reddit style news site

External user’s goal:
Users want to post, read and comment on news stories.

Site owner's goal:
The site owner would like to create a discussion community.

Potential features to include:

  • Posts can be up/downvoted
  • Comments can be left on a post
  • Time/Date of posting
  • Topic groupings/categories

Amplification (craftsmanship)

Front-End Design

Information Hierarchy
  • Semantic markup conveys structure; information is organised and easy to find
  • All resources are easy to find; navigation is intuitive
  • Information is prioritised appropriately
User Control
  • Interactions produce a positive response via layout, colour, clear navigation and feedback
  • No aggressive pop-ups/autoplay; user initiates media
  • Non-existent routes redirect back to the main page
  • Don’t ask for info you already have (e.g., logged-in email)
  • Progress indicators and transaction feedback shown
  • User/data errors are reported clearly
Consistency & Confirmation
  • Design and interactivity are consistent across sections and data operations/reports
  • Actions are confirmed; feedback is given at all times
Accessibility
  • Clear conformity to accessibility guidelines across pages/interactivity
  • Any deviations are identified and explained (comments/README)

Development & Implementation

Clean Code & Naming
  • Files/classes/functions/variables are descriptive and consistent
  • No spaces; lower-case file/dir names for cross-platform compatibility
  • Attributes/CSS rules/variables/function names follow standards and are meaningful; app URLs are consistent
File Structure
  • Files grouped by type (assets, css, js, apps, etc.)
  • Clear separation between custom and external code (e.g., libraries folder)
  • Consistent, descriptive naming without spaces/capitals
Readability
  • Consistent indentation; no repeated blank lines >2
  • IDs/classes/functions/variables indicate purpose
  • Code split into well-defined, commented sections
  • Semantic HTML; HTML/CSS/JS/Python separated and linked appropriately
  • CSS linked in head; non-trivial JS at end of body (or head if needed)
Defensive Design & Comments
  • Validate inputs (presence/format/range)
  • Handle internal errors gracefully; notify users
  • Custom code files include clear, relevant comments
Compliance & Robustness
  • HTML passes W3C; CSS passes Jigsaw; JS passes linter; Python conforms to PEP8 (or stated style)
  • No logic errors; user errors handled; failed API calls handled gracefully; inputs validated
  • Navigation never breaks site; no broken links; no console errors

The entire design is implemented, providing an excellent solution and addressing security considerations.

Real-world Application

  • Use understandable, site-specific content (no Lorem Ipsum)
  • External links open in new tabs
  • Final application aligns with initial user stories
  • Testing procedures comprehensive with good coverage; followed; errors corrected or documented

Security & Framework Conventions

  • Secrets in env/.gitignore; never committed
  • Login-required functionality restricted to logged-in users
  • Appropriate user permissions and access levels
  • Django conventions followed (Templates, Apps, Models, Views; logic placed per MVC(T) best practice)
  • Configuration/settings well-organised

Data & Configuration

  • Data fully modelled; matches schema (schema documented)
  • Datastore config in a single, easily changed location
  • Data well-structured with clear relationships; shared across apps (no duplication)
  • All CRUD present, working, and reflected immediately in the Front-End
  • Dependencies/config kept up to date; separate versions/branches where relevant; datastore not directly accessible without code

Testing Evidence

  • Comprehensive tests with clear evidence in commits; errors corrected or documented
  • Navigation/back-forward never breaks site; no broken links; no console errors
