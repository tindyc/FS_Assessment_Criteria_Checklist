Portfolio Project 5: E-commerce Applications

Project purpose:
In this project, you'll build a Full-Stack site based on business logic used to control a centrally-owned dataset. You will set up an authentication mechanism and provide paid access to the site's data and/or other activities based on the dataset, such as the purchase of a product/service.

You will also need to create a dummy social media product page, as demonstrated in the learning material. Since Facebook often deletes dummy pages, please take screenshots after you have created the page and include them in your README.md file.

If you prefer not to use or create a Facebook account, even a fake one, mockups (PNG & Balsamiq) are provided. Modify them with content to match your product and include the images in your README.md file.

Main Technologies:
HTML, CSS, JavaScript, Python + Django
Relational database (MySQL or Postgres recommended)
Stripe payments

Project Ideas

Project Idea 0
Bring your own idea(s) to life, based on providing value to users to address a specific factual or imagined need. Use the relevant project assessment criteria as a guide to the minimum required functionality.

Project Example Idea 1
Build a fitness subscription application

External user’s goal: To join a fitness community and purchase exercise plans and merchandise.

Site owner's goal: Build an active community around the product based on subscription and individual payment models; sell exercise plans, nutrition plans, and products.

Primary areas of functionality:

  • Community: Member updates, user profiles mapped to plans, product reviews
  • E-commerce: Subscriptions + one-off purchases; auth & roles for users/admins
  • SEO & Marketing: Employ SEO, social media and email strategies
  • Authentication/authorisation for subscribers and admins

Project Example Idea 2
Sell your graphic design services

External user’s goal: Purchase custom graphics.

Site owner's goal: Earn money via freelance design work.

  • Showcase portfolio + testimonials
  • Order form (type/size/brief), auto quote preview (client-side), final price on server
  • Owner dashboard to view orders and upload deliverables
  • SEO, social, email marketing strategies

Assessment Criteria — Distinction

All Pass & Merit met + publishable quality, pro-grade UI/interaction, no logic errors, justified deviations, craftsmanship, original work. SEO/Marketing and Security evidenced; Django conventions followed; robust design, data & testing; no broken links; no console errors.

Front-End Design

Information Hierarchy
  • Semantic structure; organised, easy-to-find information
  • Resources easy to find; intuitive navigation
  • Information prioritised appropriately
User Control
  • Positive interaction via layout/colour/navigation/feedback
  • No aggressive pop-ups/autoplay; user-initiated media
  • 404/unknown routes redirect back to safety
  • Don’t ask for data you already have
  • Progress indicators & transaction feedback
  • User/data errors reported clearly
Consistency, Confirmation & Accessibility
  • Consistency across pages, interactivity & reporting
  • Confirm actions; constant feedback
  • Accessibility conformance across pages & interactions; deviations explained

Development & Implementation

Clean Code & File Structure
  • Consistent naming; no spaces; lower-case dirs; meaningful identifiers; consistent URLs
  • Group files by type; clear separation of custom vs libraries; descriptive names
  • Well-commented sections; consistent indentation; no repeated blank lines
Compliance & Robustness
  • HTML (W3C), CSS (Jigsaw), JS (linter), Python (PEP8 or stated) pass checks
  • No logic errors; handle user errors; handle failed API calls gracefully; inputs validated
  • Back/forward never breaks site; no broken links; no console errors

Real-World Application, SEO/Marketing & Security

  • Real content (no Lorem Ipsum)
  • External links open in new tabs; aligned to user stories; comprehensive testing with evidence in commits
  • SEO & marketing strategies evident and documented
  • Secrets in env/.gitignore; login-required features gated; permissions appropriate

Django Conventions

  • Templates, Apps, Models, Views used correctly; logic placed per MVC(T) best practice
  • Configuration & settings well-organised

Data & Config

  • Data fully modelled; matches schema; config centralised & changeable
  • Well-structured entities & relationships; shared data across apps (no duplication)
  • All CRUD works and reflects immediately in the Front-End; deps/config up to date; datastore not directly accessible

Testing Evidence

  • Comprehensive coverage; followed; errors fixed or documented; navigation resilient
